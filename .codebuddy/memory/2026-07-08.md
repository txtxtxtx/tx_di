## AOP 拦截器链未初始化的根因与修复

**现象**：`tx_di_core` 的 `test_aop_intercept_macro_end_to_end` 失败，panic "[di] 拦截器链未初始化"。
`#[component(intercept(...))]` 生成的 `init` 覆写把拦截链按 `Arc::as_ptr(&comp) as usize` 存入全局 HashMap（`tx_di_core::aop::set_interceptor_chain`），`#[intercept]` 方法用 `self as *const Self as usize` 取出——**key 本身是对的**（诊断显示两端 key 一致）。

**真正根因**：`tx_di_core::lifecycle::App::ins_run` 用 `tokio::spawn` 把整个 `App::run`（含 `init` 阶段）放入后台 task，然后**立即**返回 `Arc<App>`。于是调用方在 `init` 阶段执行前就调用了被拦截的方法，链尚未注册（`[DIAG init]` 日志排在 panic 之后，证明 init 在方法之后才跑）。

**修复**（`tx_di_core/src/lifecycle.rs` 的 `ins_run`）：在返回前先同步完成 `App::init(&app)?` + `App::async_init(&app).await?`，仅把长期后台任务 `App::comp_run` 放进 `tokio::spawn`。`ins_run` 签名不变，插件/示例无感知。

**附带清理**：删除了 `tx_di_macros/src/intercept_macro.rs` 与 `src/codegen/intercept.rs` 中排查时遗留的 `eprintln!("[DIAG ...]")` 调试输出。

**验证**：`cargo test -p tx-di-core --test test_component` 全部 67 个用例通过；`cargo check --workspace` 通过（仅 examples 既有 warning）。
