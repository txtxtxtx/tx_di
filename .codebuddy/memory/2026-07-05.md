#### 重要发现
- `TRAIT_IMPL_MAP` 是全局静态，跨测试调用 `BuildContext::new()` 时会持续累积 trait 条目，因此不能断言固定长度
- 生命周期钩子的自由函数名 (`app_init`, `shutdown` 等) 必须放在独立模块中以避免同名冲突

#### 生命周期钩子设计发现
- `#[component(app_init)]` → 自由函数 `fn app_init(comp: Arc<Self>, app: &Arc<App>) -> RIE<()>`
- `#[component(app_async_init)]` → 自由函数 `fn app_async_init(...) -> BoxFuture<RIE<()>>`
- `#[component(shutdown)]` → 自由函数 `fn shutdown(&self)` 
- 每个钩子的自由函数名都相同 (如 `app_init`)，因此同一模块中只能有一个组件使用相同钩子
- 测试中通过 `BuildContext::build_and_run()` 验证了 init/async_init 钩子
- 通过 `App::shutdown()` 直接调用验证了 shutdown 钩子
- 通过 `App::ins_run()` 验证了完整生命周期链

#### 技术问题修复

## TRAIT_IMPL_MAP 移入 Store 重构

### 背景
全局 `TRAIT_IMPL_MAP` (`LazyLock<DashMap>`) 导致跨 `BuildContext::new()` 调用时 trait 条目不清理累积，也违反了 Store 的自包含原则。

### 修改方案
将 trait 实现映射表从全局静态迁移到 `Store` 实例字段：

#### 修改的文件
1. **`store.rs`** — `Store` 新增 `trait_impls: TraitImplMap` 字段，初始化为空 `DashMap`
2. **`store.rs`** — 移除全局 `TRAIT_IMPL_MAP` static
3. **`store.rs`** — `inject_trait_from_store` / `inject_all_traits_from_store` 改用 `store.trait_impls`
4. **`topology.rs`** — `topo_sort` 增加 `trait_impls: &TraitImplMap` 参数
5. **`lifecycle.rs`** — `auto_register_all()` 填充 `self.store.trait_impls`，而非 `TRAIT_IMPL_MAP`
6. **`lifecycle.rs`** — 传递 `&self.store.trait_impls` 给 `topo_sort`
7. **`lifecycle.rs`** — `debug_registry()` 临时构建 `TraitImplMap` 供 `topo_sort` 使用
8. **`lib.rs`** — 从 `pub use` 中移除 `TRAIT_IMPL_MAP`
9. **`di_bench.rs`** — `topo_sort` 调用处传递空的 `TraitImplMap`

### 关键设计决策
- `Store::from_dashmap()` 创建的 Store 的 `trait_impls` 为空（需后续手动填充）
- `Store::into_inner()` 只返回组件 DashMap，不包含 trait_impls（保持向后兼容）
- 每个 `BuildContext::new()` 调用独立填充 `store.trait_impls`，无全局污染

### 测试验证
- 62 tests passed, 0 failed
- `test_inject_all_traits_from_store` 现在每个 Store 独立，不跨测试累积

## 插件框架 API 迁移（2026-07-05）

### 背景
用户框架（tx-di-core + tx-di-macros）已更新到新 API（`#[derive(Component)]` + 自由函数回调），
移除了旧的 `#[tx_comp(...)]`、`CompInit` trait、`InnerContext` 类型。
日志插件（tx_di_log）已先期更新，其余 5 个插件需要迁移。

### 前置改动
1. `async_method!` 宏从返回 `impl Future + Send` 改为返回 `$crate::BoxFuture`（`Pin<Box<dyn Future + Send>>`）。
2. **最终移除 `async_method!` 宏**：用户直接写 `async fn ... -> RIE<()>`，`#[derive(Component)]` 生成代码自动通过 `Box::pin(async move { callback().await })` 包装为 `BoxFuture`。

### 迁移的 5 个插件

| 插件 | 组件 | 关键变更 |
|------|------|---------|
| **tx_di_file** | FileConfig, FilePlugin | conf+init / async fn app_async_init |
| **tx_di_sa_token** | SaTokenConf, SaTokenPlugin | conf+init / fn init |
| **tx_di_axum** | WebConfig, WebPlugin | conf+init / async fn app_async_init + async fn app_async_run; 修复 bound/mod.rs 中 `ComponentDescriptor`→`Component`、`IE`→`AppError` |
| **tx_di_toasty** | ToastyConfig, ToastyPlugin | conf+init / async fn app_async_init |
| **tx_di_job** | JobConfig, JobPlugin | conf+init / async fn app_async_init + async fn app_async_run |

### 发现的问题及修复
1. **lifetime 问题**：`BoxFuture` 要求 `'static`，`app: &Arc<App>` 引用不满足。修复：`lifecycle.rs` 中 `app.clone()` 传入 async block，用户回调签名改为 `app: Arc<App>`。
2. **DepsTuple 必须在 scope**：`#[derive(Component)]` 生成的代码调用 `Deps::resolve(store)`，需要 `DepsTuple` trait 在 scope。
3. **async_method! 宏移除**：包装逻辑移入 `lifecycle.rs` 生成代码，用户写普通 `async fn`。

### 其他插件状态
- `tx_di_sip`、`tx_di_can`、`examples/` 中的项目尚未迁移，仍使用旧的 `tx_comp`/`CompInit`，编译会失败。
- 工作区 `cargo check --workspace` 会因此报错，但 6 个目标插件各自 `cargo check -p` 均通过。
