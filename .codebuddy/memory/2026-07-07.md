## tx_di_can 上位机需求梳理（2026-07-07）

- 用户目标：为嵌入式诊断工程师做 CAN 诊断**上位机**（带界面的工具）。
- 现状盘点：`tx_di_can` 是**后端协议引擎库**（adapter/ISO-TP/UDS/flash），但：
  - 致命缺口：SocketCAN/PCAN 适配器**无后台接收任务**，真实硬件收不到帧（只有 SimBus 回环可用）。
  - `plugin.rs` 仍用废弃的 `#[tx_comp(init)]`/`CompInit`/`InnerContext`/`BoxFuture`，需迁移到 `#[derive(Component)]`，目前大概率编译不过。
  - 缺 CAN-FD ISO-TP、DBC 信号解码、DTC/DID 描述库、帧过滤/统计、时间戳（恒0）、录制/回放、前端 UI。
- 产出：`plugins/tx_di_can/REQUIREMENTS.md` 第一版需求文档（含现状对照表、必修技术缺口、前端三方案 Tauri/egui/Web、MVP 范围、待确认问题）。
- 待用户确认：UI 框架、目标硬件/OS、协议范围（仅UDS / +FD / +J1939...）、MVP 优先级。

## CAN 上位机范围确认（2026-07-07，用户已答）
- 决策：Tauri（Rust 后端 + Vue3/TS 桌面 App） / PCAN(PEAK)+Windows 优先 / 含 CAN-FD 与 FD 版 ISO-TP / 监控+诊断+刷写 完整 MVP 同批。
- 文档已升级为 `plugins/tx_di_can/REQUIREMENTS.md` v2（架构、缺口、分阶段计划）。
- 关键核实结论（纠正了 v1 的偏差）：
  - PCAN 适配器 FFI 只有 `CAN_Write`/`CAN_Open`/`CAN_Init`/`CAN_Close`，**缺 `CAN_Read`/`CAN_ReadFD` 且无后台接收任务** → PCAN 现只能发不能收（致命缺口具体形态）。`close()` 的 `tasks`/`running` 是未用的接收脚手架。
  - `plugin.rs` 的 `start_rx_loop` 接收回路逻辑其实存在，但挂在废弃的 `CompInit::async_init` 上，迁移到 `#[derive(Component)]` 后需重接。
  - `tx_di_can` 已具备 `CanFdFrame`/`send_fd` 与 SocketCAN FD 发送；缺的是 **FD 版 ISO-TP（64字节+escape）**。
- 建议 Tauri 应用目录：`plugins/tx_di_can/app/{src-tauri,src}`；通信用 Tauri command + `app.emit` 转发 `CanEvent`。
