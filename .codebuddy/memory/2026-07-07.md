## tx_di_can FD ISO-TP P0-3 完成（2026-07-07 深夜）

- `isotp.rs` 已实现完整 **CAN-FD ISO-TP（ISO 15765-2:2016）** 收发路径，`IsoTpConfig.is_fd` 控制 Classic/FD 分支；`plugin.rs` 已按 `enable_fd` 把 FD 帧经 `tokio::select!` 转发到 `CanEvent::FdFrameReceived`，`send/recv` 自动分流。
- **关键修复（核心 bug）**：FD 首帧（FF）解码时，原代码用 `hi = data[0] & 0x0F; if hi == 0 { escape }` 判定 escape，**错误**。因为普通 FD FF 当总长 ≤ 255 时 `data[0] = 0x10 | 0 = 0x10`，`hi==0` 被误判为 escape，导致 `data[1] != 0x00` 直接报错、永不发 FC，表现为"等待 FC 超时"。正确判据应为 `data[0] == 0x10 && data[1] == 0x00`（FD FF escape 是 4 字节大端长度）。
  - 顺带修正：测试的 `payload` 原来写成 `(0..len as u8).collect()`，对 `len>255` 会截断（如 500→244 字节），改为 `(0..len).map(|i| i as u8)` 生成真实长度。
  - `send_fd_single` 去掉不可达的 63..=4095 单帧 escape 分支（CAN-FD 帧上限 64 字节，单帧最多 62 字节，>62 由 `send_fd` 路由到多帧）。
- **测试要点（SimBus 回环的坑）**：SimBus 用 `tokio::broadcast`，只投递订阅之后发出的帧。单元测试里 ECU 接收方**必须先在发送方 `send` 之前 `subscribe_fd()`**，否则首帧被丢弃导致"等待 FC 超时"。FD 测试用 `recv_on_fd(ecu_rx, ...)`（预订阅 receiver）规避。

## P2 互通加固完成（2026-07-07 深夜，0 警告全测过）

四项 P2 加固全部落地并 `cargo test` 通过（tx_di_sip 100 / tx_gb28181 51 / gb_cams 7 / gb_dev 6 / common 3，0 warnings）：

1. **SDP NAT 重写**：`common/tx_gb28181/src/sdp.rs` 新增纯函数 `rewrite_connection_address(sdp, new_ip) -> String`，重写会话级/媒体级 `c=` 行与 `o=` 行地址（自动推导 IP4/IP6），附 4 个单测。L2a 侧在 `plugin_tail.rs` 新增私有 `media_ip()`（优先级 `nat_external_ip` > `media.local_ip`(非未指定) > `sip_ip`），4 处出网 SDP 构建点统一改用之；`Gb28181ServerConfig::MediaConfig` 新增 `#[serde(default)] nat_external_ip: Option<String>`（向后兼容）。
2. **in-dialog 关联助手**：`plugins/tx_di_sip/src/dialog.rs`（新模块，lib.rs 导出 `DialogKey`/`InDialogTable`）。`DialogKey::from_request(&Request)` 用 `HeadersExt` 的 `call_id_header()/from_header()/to_header()` 的 `.value()`（注意：typed header 的 `Display` 会加 `Call-ID:`/`From:` 前缀，必须取 `.value()` 才得原始头值），解析 `tag=` 参数得到 Call-ID+From-Tag+To-Tag；`InDialogTable<T>` 按完整键精确查表，并支持以「半键」(to_tag=None) 回退关联到已建对话。附单测。
3. **transport 默认 Both**：`tx_di_sip::config::SipTransport` 默认从 `Udp` 改为 `Both`（国标场景兼容老设备 UDP 与 NAT 友好 TCP）；更新 3 处断言默认值的测试（含新增双向 TLS 配置解析测试）。
4. **双向 TLS**：`TlsConfig` 新增 `ca_certs`(校验对端/mTLS)、`client_cert`/`client_key`(作为 TLS 客户端发起 mTLS)；`comp.rs::build_tls_transport` 读取这些 PEM 文件并传入 rsipstack `TlsConfig`。

- 关键坑：`rsipstack` typed header 的 `call_id_header()` 返回 `Result`（不是 `Option`），需 `.ok()?`；`.value()` 返回 `&str`（原始头值），`.to_string()` 才是带 `Call-ID:` 前缀的 Display 串——提取 tag 必须用 `.value()`。
